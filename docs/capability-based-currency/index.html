Capability Money Network (CMN)

A fixed-supply, indivisible, capability-based money system in which each coin is either claimed or unclaimed, spending transfers control to fresh public keys, and long-inactive claimed coins return to the unclaimed reserve for future circulation.

1. Purpose

This specification defines a capability-based monetary system built from discrete indivisible coins rather than account balances.

The purpose of the system is to enable:

  • fixed-supply money per context
  • capability-based control of individual coins
  • bootstrap distribution through heavily rate-limited free claiming
  • payment by transferring coin control to public keys supplied by the payee
  • return of long-inactive claimed coins to the unclaimed reserve
  • circulation reinforcement through spend-triggered release of unclaimed coins

This model treats money as discrete transferable capabilities over coins rather than as mutable balances in named accounts.

2. Design goals

The system is designed to satisfy the following properties:

  1. Minimal state model.
    Every coin must have exactly one of two states: claimed or unclaimed.
  2. Fixed supply.
    No new coins are created after genesis. Coins only move between claimed and unclaimed control states.
  3. Indivisible units.
    Each coin is one discrete minimal denomination unit and cannot be subdivided.
  4. Capability-based control.
    Spend authority must follow possession of the private key corresponding to the current controlling public key.
  5. Circulation preference.
    Long-inactive claimed coins must not remain permanently controlled forever.
  6. Bootstrap viability.
    Early participation must be possible through limited direct claiming of unclaimed coins.
  7. Spend-driven maturity.
    As the system becomes active, spending must become the primary path by which dormant reserve supply re-enters circulation.

3. Non-goals

This specification does not require:

  • account balances as the primitive money model
  • fractional coin units
  • inflationary issuance after genesis
  • permanent ownership without continued participation
  • domain-specific monetary authority

4. Terminology

Context
A monetary scope with one fixed coin supply and one shared rule set.

Coin
A single indivisible monetary unit with a unique identifier.

Claimed
A coin state meaning some public key currently controls the coin.

Unclaimed
A coin state meaning no public key currently controls the coin.

Controller
The holder of the private key corresponding to a claimed coin’s controlling public key.

Claim
A transition assigning an unclaimed coin to a public key.

Spend
A transition transferring one or more claimed coins from current controllers to destination public keys.

Payment Request
A message in which a payee supplies one destination public key per requested coin.

Unclaimed Reserve
The current set of all coins in the unclaimed state.

Inactivity Expiry
The rule by which a claimed coin becomes unclaimed after sufficient inactivity.

5. System model

The system consists of:

  • one or more monetary contexts
  • a fixed set of coin identifiers per context
  • a two-state per-coin model
  • a claim mechanism
  • a spend mechanism
  • an inactivity expiry rule
  • a spend-triggered reserve return rule
  • a deterministic conflict resolution rule for competing spends

A coin is controlled when claimed and uncontrolled when unclaimed. The network’s job is to determine which public key, if any, currently has spend authority over a given coin.

6. State model

6.1 Allowed states

A coin has exactly two states:

  • claimed
  • unclaimed

No other coin states exist.

6.2 Meaning of claimed

A claimed coin has one active controlling public key. Whoever holds the corresponding private key has the exclusive right to spend that coin.

6.3 Meaning of unclaimed

An unclaimed coin has no active controlling public key. No current holder has spend authority over it until it is claimed again.

6.4 Non-states

The following are not states:

  • spent
  • expired
  • reclaimed
  • rewarded
  • minted

These are events, transition causes, or policy results. The coin itself still ends in either claimed or unclaimed.

7. Coin model

7.1 Minimum conceptual structure

A coin conceptually contains at least:

  • coin_id
  • state
  • controller_public_key when claimed
  • last_claimed_at
  • last_spent_at

7.2 Notes

  • coin_id uniquely identifies the coin inside its context
  • controller_public_key exists only when the coin is claimed
  • last_claimed_at records the latest transition from unclaimed to claimed
  • last_spent_at records the latest spend transfer while claimed
  • exact encoding is implementation-defined

8. Supply and genesis

8.1 Fixed supply

Supply is fixed per context. No post-genesis rule may create additional coins. The system only changes whether specific existing coins are claimed or unclaimed, and who controls claimed coins.

8.2 Genesis patterns

A context may begin in either of the following patterns:

  • all coins initially unclaimed
  • some subset initially claimed and the rest unclaimed

The preferred bootstrap pattern is that a large reserve of coins begins unclaimed.

9. Claiming

9.1 Claim transition

A claim assigns an unclaimed coin to a public key and changes the coin from unclaimed to claimed.

9.2 Claim input

A claim includes at least:

  • target coin_id
  • claimant public_key
  • any required rate-limit or anti-abuse proof

9.3 Claim result

If valid, the network sets:

{
  "state": "claimed",
  "controller_public_key": "claimant public key",
  "last_claimed_at": "now"
}

9.4 Bootstrap freebie claiming

The system may begin with freebie claiming from the unclaimed reserve. This is intended only as a bootstrap distribution mechanism.

The design intent is:

  • new participants can enter the economy without prior holdings
  • one actor cannot efficiently drain the reserve
  • direct claiming becomes less strategically meaningful as spend activity grows

9.5 Claim rate limiting

Claiming should be heavily rate-limited. The exact mechanism is implementation-defined, but its purpose is to make direct claiming increasingly marginal compared to obtaining coins through actual economic circulation.

10. Payment requests

10.1 Payee-generated destination keys

A payee generates one or more fresh key pairs and sends the payer one destination public key per requested coin.

10.2 Payment request contents

A payment request should contain at least:

  • requested amount in coin count
  • one destination public key per requested coin
  • optional request identifier
  • optional expiry time

10.3 Payment model

A payment is performed by the payer spending coins they currently control and rebinding those coins to destination public keys supplied by the payee.

11. Spending

11.1 Spend transition

A spend transfers one or more claimed coins from current controllers to new destination public keys. The coin remains claimed throughout; only the controlling key changes.

11.2 Spend input

A spend includes at least:

  • one or more input coin identifiers
  • one destination public key per transferred coin
  • signatures from the current controllers of all input coins

11.3 Spend validity

A spend is valid only if:

  • every input coin exists
  • every input coin is currently claimed
  • every input coin is controlled by the corresponding signing key
  • no input coin has already been consumed by a canonical competing spend
  • the number of input coins and destination keys conforms to the context’s transfer rules

11.4 Spend result

For each valid transferred coin, the network updates:

{
  "state": "claimed",
  "controller_public_key": "new destination public key",
  "last_spent_at": "now"
}

The previous controller loses spend authority immediately once the spend becomes canonical.

12. Inactivity expiry

12.1 Rule

A claimed coin does not remain claimed forever. If it remains inactive long enough, it returns to the unclaimed reserve.

The default inactivity expiry is one year.

12.2 Expiry conditions

A claimed coin becomes unclaimed if either of the following is true:

  • now - last_spent_at > 1 year
  • for a claimed coin never yet spent, now - last_claimed_at > 1 year

12.3 Expiry result

When inactivity expiry occurs, the network updates:

{
  "state": "unclaimed",
  "controller_public_key": null
}

The prior holder loses spend authority. The coin is then part of the unclaimed reserve again.

13. Unclaimed reserve

13.1 Definition

The unclaimed reserve is simply the current set of all unclaimed coins in a context.

13.2 Sources of reserve

Coins enter the unclaimed reserve by either of the following paths:

  • they were never yet claimed since genesis
  • they were previously claimed but later returned to unclaimed by inactivity expiry

14. Spend-triggered reserve return

14.1 Principle

When spending occurs, the network may release some coins from the unclaimed reserve back into circulation.

This makes active spending a mechanism by which dormant supply re-enters the economy.

14.2 Default policy shape

For a spend of amount S, the network may return up to a bounded fraction of S from the unclaimed reserve.

Example policy:

  • return up to 10% of S
  • actual return depends on currently available unclaimed reserve
  • if insufficient reserve exists, return less
  • if no reserve exists, return zero

14.3 Example formula

reward = min(unclaimed_reserve_available, floor(spend_amount * 0.10))

14.4 Routing of returned reserve coins

The context must define how returned reserve coins are routed. A clean default is that they are attached to spend outputs so they re-enter circulation through actual payment flow rather than idle claiming.

15. Conflict resolution

15.1 Double spend condition

A claimed coin may be the subject of multiple conflicting spend attempts. The network must choose exactly one canonical result.

15.2 Deterministic canonical selection

If two or more valid competing spends reference the same claimed coin, all nodes must apply the same deterministic conflict resolution rule and arrive at the same canonical winner.

The exact deterministic rule is implementation-defined.

16. Participation

16.1 Domain-agnostic participation

Any domain may participate in the system. A domain is not the monetary authority by itself. The monetary rules belong to the context, not to a single host or operator.

16.2 Holder model

A holder participates by generating key pairs, receiving claimed coins to their public keys, and spending claimed coins using the corresponding private keys.

17. Security considerations

Implementations must consider at least:

  • private key protection for claimed coins
  • fresh-key use for receiving payments
  • claim abuse prevention
  • conflict resolution consistency across the network
  • replay resistance for spend messages
  • validation of payment request structure
  • rate-limit circumvention attempts
  • reserve farming through abusive low-value activity

The system assumes that control is defined by possession of the correct private key and that private keys remain under sole control of their holders.

18. Economic character

18.1 Monetary nature

This is not an account-balance money system. It is a capability money system built from discrete indivisible coins.

18.2 Renewable control rather than eternal possession

Claimed control is durable but not eternal. If a coin remains inactive for too long, it returns to the unclaimed reserve.

18.3 Bootstrapping and maturity

The economy evolves in two broad phases:

  • Bootstrap phase: heavily rate-limited direct claiming helps participants enter the system
  • Mature phase: spending becomes the dominant mechanism by which reserve supply returns to active circulation

19. Example flow

19.1 Bootstrap claim

  • coin C1 is currently unclaimed
  • a participant generates key pair K1
  • they submit a valid claim for C1 with K1.public
  • C1 becomes claimed under K1.public

19.2 Payment

  • a payee generates fresh public keys P1, P2, and P3
  • the payer controls claimed coins C1, C2, and C3
  • the payer signs a spend transferring those coins to P1, P2, and P3
  • the payee now controls those claimed coins through the corresponding private keys

19.3 Inactivity expiry

  • coin C7 remains claimed but unused for more than one year
  • the network applies inactivity expiry
  • C7 becomes unclaimed
  • C7 is again part of the unclaimed reserve

19.4 Spend-triggered reserve return

  • a payer spends 100 claimed coins
  • the context allows up to 10% reserve return
  • the network checks the available unclaimed reserve
  • if available, up to 10 additional coins may be pulled from the reserve into the transaction outputs

20. Minimal conformance requirements

A conforming implementation must:

  1. define a fixed coin supply per context
  2. model every coin with exactly two states: claimed and unclaimed
  3. allow valid claims from unclaimed to claimed
  4. allow valid spends that transfer claimed control to new public keys
  5. apply inactivity expiry from claimed to unclaimed
  6. define a deterministic rule for conflicting spends
  7. define a claim rate-limit policy
  8. define a spend-triggered reserve return policy, even if the configured return amount is zero

21. Summary

This specification defines a fixed-supply, indivisible, capability-based money system with only two coin states: claimed and unclaimed.

Coins begin either claimed or unclaimed, may be directly claimed under strong rate limits, may be transferred by spending to fresh public keys, and return to the unclaimed reserve after long inactivity.

The key monetary principle is:

a coin is either currently controlled by someone or currently controlled by no one, and circulation is reinforced by returning unclaimed reserve supply through spending.
